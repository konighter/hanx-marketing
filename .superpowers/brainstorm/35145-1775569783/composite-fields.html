组合聚合字段 (Composite Fields) 的展现探索

针对图片、五点描述、长宽高尺寸等特殊的“原子组合”属性，由于它们在 Schema 中通常被打平得很难看（如 item_dimensions.length.value），我们需要一种能兼顾体验的展现方式。

A

Schema 驱动的类型化组件 (Type-based Components)

完全废弃通用 el-input。当系统在 Schema 中识别到特定的结构化标识（例如：x-hz-component: "image-upload" 或字段名为 item_dimensions），前端动态表单直接挂载一个专门针对该字段定制的 Vue 组件（内部处理了亚马逊那种嵌套很深的 JSON 结构和单位换算）。

表现

五点描述会直接呈现为“带有添加/排序按钮的动态列表组件”，而不是5个平铺的普通文本框。

B

表单组群化 (Field Grouping & Layout Wrapping)

依然维持通用的单字段绑定，但是在后端的配置生成阶段（SchemaService 中），我们将 item_dimensions 下的“长、宽、高、重”自动划分为同一个“聚合 Group”。前端遇到带有聚合特征的 Group 时，渲染成一个自带单位（Unit Selector）的卡片盒子把它们包起来。

表现

相对简单，只需增加排版逻辑，不改变动态表单组件底层对 “扁平 Key” (flat keys) 读写的本质，避免编写大量平台专属定制组件。

C

ERP 主数据直接强行接管 (ERP Global Takeover)

这也是 B 路线（ERP核心驱动）最彻底的做法。凡是通用的复杂字段（图片、尺寸、包装重量），在动态表单里干脆 完全不渲染 Amazon 的 Schema 版本。
而是统一让用户在最顶端的『ERP主数据』版块进行填写，然后由后端系统负责把长宽高包装成 Amazon 恶心的 {length:{value:1, unit:'cm'}} 结构发走。

表现

用户界面极其清爽，但如果不同平台/不同品类对图片像素、尺寸单位有独特验证规则，这套主数据映射就有很大挑战。
