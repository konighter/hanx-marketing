商品多品类录入机制探索 (TikTok / Shopee)

我们在 ERP 中需要一套能兼顾多平台、多类目差异的结构，这里有 3 个不同的方向：

A

平台优先强管控 (Shopee / TikTok 风格)

用户先选择目标平台及对应类目，表单完全依照该平台获取的 Schema 动态生成对应的必选项和变体网络。

优势

数据结构 100% 对齐平台，绝不会出现漏填属性导致发布报错，变体组与约束最为直观。

劣势

多平台需要重复制单，比如同样卖一件衣服，TikTok 填一遍，Shopee 再填一遍。

B

基于内部主数据的自动映射 (ERP 理想模式)

用户基于 ERP 内部预设类目填写一张极少部分通用属性的大表单（如标题、公共图片、基本变体）。随后由系统建立强大的 Mapping 字典向多平台映射。

优势

“一键刊登”的极致体验，一品多发，操作成本最低。

劣势

技术维护成本极高，需要预先建立并长期维护底层类目映射、属性字典、以及平台间复杂的约束转换差异。

C

主被动结合：双栏/分面补充录入 (推荐混合态)

左侧为基础主数据表单，右侧/下方为多平台 Tab 页。用户先在一个通用视图里填写共享池数据，点击对应平台 Tab 时，动态加载那些该平台特有的、未被映射的必填属性片段 Schema。

优势

兼顾了一键多发与各个平台差异性补充。是目前市面上大型店群 ERP 最主流的做法（兼顾灵活性与准确度）。

劣势

UI 复杂度较高，对动态表单的数据流转拼接和 Schema 隔离要求较高。
