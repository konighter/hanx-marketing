方案 B：刊登任务处理时序图

展示从同步接收请求到异步平台回调的完整数据流

刊登执行流程 (Sequence Diagram)
User / UI
Service (Sync)
Listing Task DB
Task Queue / MQ
Executor (Async)
Platform API
1. Submit Listing
2. persist() PENDING Task
3. dispatch(taskId)
4. Return taskId
ASYNC PROCESSING START
5. Consume Task
6. Platform.post()
7. Success (ASIN/ItemID)
8. Update SUCCESS + Sync Product表

关键流程说明：

  1. 快速响应：Controller 只需要完成最基础的参数校验，并将记录存入任务表，然后立即返回 200 给前端。用户不需要等待翻译和平台上传。
  2. 任务完整性：任务表中会保存所有原始的 attributes JSON，这保证了如果刊登失败，我们可以“原封不动”地让用户在前端点击编辑，重新加载这些数据。
  3. 最终同步：当平台返回成功的标志（如 Amazon 的 ASIN）后，由 Executor 负责将数据正式写入“平台商品表”。
