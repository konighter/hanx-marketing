任务引擎分界设计 (Boundary Design)

明确引擎与业务子模块的职责边界，确保低耦合

引擎职责 (Engine Responsibilities)
  • 任务的 CRUD 与 持久化
  • 状态流转 (PENDING -> RUNNING -> ...)
  • 定时扫描与 多线程分发
  • 异常捕获、重试计数、日志记录
  • 父子任务的 完成度检测 与级联更新
子模块职责 (Module Responsibilities)
  • 任务 参数组装 (如 DateRange, ShopId)
  • 实现 执行逻辑 (Handler 内部逻辑)
  • 调用引擎 提交任务 (调用 submitTask)
  • 不直接操作 ads_sync_task 表
  • 不负责状态更新，仅返回 执行结果 给引擎

关键接口定义 (API Design)

// 引擎提供的服务接口
public interface AdsTaskEngine {
    // 提交任务请求，返回任务ID
    Long submitTask(AdsTaskSubmitRequest request);
    
    // 内部调用：更新任务状态（子模块不可见）
    // void updateTaskStatus(Long taskId, AdsTaskResult result);
}

// 子模块需要实现的处理器接口
public interface AdsTaskHandler {
    // 执行具体业务逻辑，返回标准化的结果对象
    AdsTaskResult execute(AdsSyncTaskDO task);
    
    // 处理器标识：如 "REPORT_DAILY"
    String getTaskType();
}

对于异步任务（如 Amazon 报表），您倾向于哪种执行模式？

A

状态反应式 (Single-Method)

引擎只要任务没完成就调用 execute。Handler 内部根据状态判断是“提交报表”还是“轮询状态”。接口最简单。

B

生命周期钩子 (Multi-Method)

Handler 显式提供 onStart, onPoll, onRetry 等方法。结构更清晰，但接口复杂度略高。
