结果回写契约 (Result Contract)

Handler 如何通知引擎任务的最终结局

核心契约：AdsTaskResult 对象

Handler 不直接操作数据库，而是通过返回一个统一的结果对象，由引擎负责后续的 DB 更新和关联逻辑。

Result 对象定义
public class AdsTaskResult {
    // 任务状态：SUCCESS, FAILED, SUBMITTED
    private String status;
    
    // 错误信息（仅 FAILED 时需要）
    private String errorMessage;
    
    // 异步 Job ID（可选，供后续轮询使用）
    private String platformJobId;
    
    // 下次执行需要的上下文（可选，更新任务的 context 字段）
    private Map<String, Object> context;

    // 静态工厂方法，让代码更易读
    public static AdsTaskResult success() { ... }
    public static AdsTaskResult failed(String msg) { ... }
    public static AdsTaskResult submitted(String jobId) { ... }
}

引擎内部的处理流程 (Engine Logic)

1. 引擎捕获到 Handler 返回的 AdsTaskResult。

2. 状态回写：将 result.status 更新到 ads_sync_task 表。

3. 元数据更新：同步 errorMessage、platformJobId 以及 context。

4. 生命周期维护：如果状态是 SUCCESS，自动记录 finishedAt。

5. 级联处理：如果是子任务完成，引擎自动检查父任务的所有子任务是否都已完成，若是则将父任务标为 SUCCESS。

异常处理 (Exception Handling)

如果 Handler 执行过程中抛出未捕获异常：

  • 引擎会自动捕获该异常。
  • 重试逻辑：retryCount++，如果未达到 maxRetries，设置状态为 PENDING 并更新 scheduledAt（退避算法）。
  • 熔断逻辑：达到最大重试次数后，强行标记为 FAILED 并记录异常堆栈。

您认为这种“基于返回对象的被动回写”模式是否足够？

A

非常合适

Handler 逻辑纯粹，完全不需要关心数据库操作，测试也更方便。

B

需要 Handler 能主动更新进度

比如 0%...50%...100% 的进度回写，可能需要 Handler 注入一个 ProgressAPI。
