高耗时/异步处理方案 (Async Execution)

解决“方法已返回，但任务仍在后台执行”的状态回写问题

方案一：阻塞式处理（推荐用于下载/入库）

Handler 的执行即任务的生命。对于高耗时任务，不要开启异步线程，而是直接在 Handler 中完成。

1. 引擎调用 onExecute()。

2. Handler 内部执行 download() + saveToDoris()（耗时 5 分钟）。

3. 执行完后返回 AdsTaskResult.success()。

4. 引擎收到返回结果，回写数据库。

优点：逻辑简单，状态一致。缺点：占用引擎的一个线程。解决方案：引擎使用专门的高并发线程池。

方案二：事件驱动/回调模式（真正的异步）

如果业务模块必须将任务派发给另一个异步服务（如另一个微服务或专用下载集群）：

异步回写时序
  1. Handler 调用异步服务，并返回 PROCESSING 给引擎。
  2. 引擎将任务标记为 PROCESSING 并挂起（不再主动捡起）。
  3. 异步服务处理完成后，通过 引擎提供的 API 进行回调： 
    taskEngine.completeTask(taskId, AdsTaskResult.success());

优点：不占引擎线程。缺点：业务模块需要感知引擎的 completeTask 接口。

对于“下载入库”这种场景，您希望 Handler 是怎么执行的？

A

阻塞执行 (推荐)

引擎分配专门的线程给 Handler，Handler 慢慢下、慢慢传，跑完返回结果就行。

B

异步回调

Handler 只管发指令，真正的下载由另一个组件异步完成，并调用引擎接口结案。
